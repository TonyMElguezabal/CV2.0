## Why

The owner wants to deploy CareerDNA to Cloudflare (existing operational home for other projects) rather than Vercel. Two rounds of hands-on verification — real `@opennextjs/cloudflare` builds plus `wrangler dev` runs against throwaway copies of the app — found concrete incompatibilities that block this today:

1. **`app/api/chat/route.ts`'s `loadIndex()`** reads `lib/rag/index.json` via `node:fs.readFileSync` at request time. Despite Cloudflare's own `nodejs_compat` docs describing a `/bundle` virtual filesystem, OpenNext Cloudflare's own Node-compat shim throws `[unenv] fs.readFileSync is not implemented yet!` for this call in practice (confirmed by running it, not just reading docs).
2. **`app/opengraph-image.tsx`** (`next/og`'s `ImageResponse`) initially *appeared* to hit the identical `fs.readFileSync not implemented` error via its default system-font loading — but this was a false negative from testing against an unpopulated incremental cache (see #5 below and design.md Decision 5). Once the cache is correctly configured and populated, this route (like the landing page) is prerendered once during `next build` — a real Node.js process with full filesystem access — and served from that cached output thereafter, never re-executing in the Worker. No code change is needed here; confirmed by a real, correctly-configured build.
3. **`proxy.ts`** (Next 16's Node.js-runtime Proxy, gating `/admin` per the accepted `admin-access-gate` capability) cannot build at all under the adapter — OpenNext Cloudflare currently rejects any Node-runtime Proxy outright (`Node.js middleware is not currently supported`), a live, unresolved gap tracked upstream at `cloudflare/workers-sdk#13755`. Reverting to the legacy Edge `middleware.ts` convention does not reliably avoid this either.
4. **`lib/content/read.ts`** (`getProfile()`, `getExperiences()`, `getFaq()`, etc.) has the identical `readFileSync`/`readdirSync`-based design as the RAG index, and is called at *request time* — not just build time — in two places that only surfaced once `/admin` and `/api/chat`'s error paths were actually exercised against a real build: `/admin`'s dashboard page calls `getExperiences()` directly, and (since `/admin` is `force-dynamic`) the root layout's `getProfile()`/`getFaq()` calls (added by the hero-laptop and chat-greeting stories) re-execute for every admin request too; separately, `/api/chat`'s rate-limited/unavailable error responses call `getProfile()` just to include the contact email/scheduling link. Both throw the same error under the adapter — because, unlike the landing page or `/opengraph-image`, `/admin` is genuinely `force-dynamic` and does not benefit from the static-assets cache in #5.
5. **The adapter's incremental cache must be explicitly configured and populated** for prerendered routes to be served as build-time output rather than re-executed live. With the default (unconfigured) cache, a raw `wrangler dev` invocation against a fresh build makes *every* route — including genuinely static, content-free ones — appear broken, which is what first suggested #2 was a required fix. Configuring the adapter's static-assets incremental cache override, and always verifying via `opennextjs-cloudflare preview` (which populates it) rather than raw `wrangler dev`, resolves this and confirms the landing page and `/opengraph-image` need no code changes at all — see design.md Decision 5.

Additionally, `@opennextjs/cloudflare@1.20.2` requires `next@">=16.2.11"`; this repo pins the one-patch-earlier `16.2.10`.

Fixing #4 surfaced a real, adjacent product question — owner-decided: the admin dashboard currently renders inside the full public root layout (hero laptop, "Ask about Jose" chat widget, footer), purely because Next's App Router root layout wraps every page. The owner chose to give `/admin` its own independent, minimal root layout (no marketing chrome) rather than keep threading precomputed content into the shared layout — this both fixes the Cloudflare crash and removes chrome that likely never belonged on an internal dashboard.

This change makes the app deployable to Cloudflare Workers by fixing all five issues, without changing any public-site-visible behavior. The two intentional behavior changes are both owner-decided: the admin login UX (a cookie-session login form replacing the native browser Basic-Auth popup, since Route Handlers — unlike Proxy — build and run fine under the adapter) and the admin dashboard no longer rendering the public marketing chrome.

## What Changes

- **RAG index loading**: `lib/rag/retrieve.ts`'s `loadIndex()` switches from a runtime `readFileSync` call to a dynamic `import()` of the generated index (bundled at build time, resolved lazily so merely importing the module doesn't require the file to already exist). No change to the build-time generation step (`lib/rag/embed.ts`, the `content-indexing-pipeline` capability) or to retrieval/generation behavior (`retrieval-grounded-generation` capability) — this is purely how the already-generated index gets into memory. `loadIndex()` becomes `async`; its one call site (`app/api/chat/route.ts`) awaits it.
- **`app/opengraph-image.tsx` needs no code change**: with the static-assets incremental cache correctly configured (below), this route is prerendered once at build time and served from cache thereafter — its `getProfile()` call and `ImageResponse`'s internal default-font lookup both execute only during `next build`'s own Node.js process, never in the deployed Worker. (An earlier attempt to fetch an explicit font via `fetch(new URL(..., import.meta.url))` was reverted — that pattern isn't supported by `next build` itself under this route's runtime target and broke the build outright, independent of Cloudflare.)
- **Admin authentication**: `/admin`'s owner-only gate moves off `proxy.ts` (Node-runtime Proxy, unsupported by the adapter) onto a cookie-session login: a login form posts to a new Route Handler that verifies the existing `ADMIN_USER`/`ADMIN_PASSWORD` credentials (same timing-safe comparison, same Upstash rate limiting) and sets a signed, httpOnly session cookie scoped to `/admin`; a protected layout checks that cookie via `cookies()` and redirects to the login form when absent or invalid. `proxy.ts` is removed. The public site is unaffected — the session cookie is admin-scoped only, consistent with the capability's "no cookie on the public site" requirement.
- **Admin gets its own root layout**: `/admin` no longer renders inside the public site's root layout. `app/layout.tsx` moves to `app/(marketing)/layout.tsx` (a route group, so the public URL structure is unchanged) alongside `app/page.tsx` moving to `app/(marketing)/page.tsx`; `app/admin/layout.tsx` becomes its own independent root layout with no hero/chat-widget/footer/structured-data. Route Handlers and metadata files (`app/api/**`, `robots.ts`, `sitemap.ts`, `opengraph-image.tsx`) are unaffected by this split (they don't render through a layout).
- **A small build-time "site config" artifact** replaces the two remaining request-time content reads: `/api/chat`'s error responses (need just the contact email/scheduling link) and `/admin`'s dashboard (needs just the ordered chapter-ID list for report queries) now import a tiny generated JSON (`{ contact, chapterIds }`) instead of calling `getProfile()`/`getExperiences()` at request time. Generated at build time (same `prebuild` step, using `lib/content/read.ts` exactly as today — that module itself is untouched, still `readFileSync`-based, still used correctly by every build-time/static consumer). This keeps the fix narrowly scoped to the two dynamic routes rather than reworking `lib/content/read.ts` globally, which would break its existing fixture-based (`contentRoot`-parameterized) test suite for no benefit — every other consumer of that module renders on static, prerendered routes and is unaffected.
- **Dependency bump**: `next` moves from the pinned `16.2.10` to `16.2.11` (the adapter's minimum supported patch; latest available 16.2.x).
- **Cloudflare/OpenNext tooling added**: `wrangler.jsonc`, `open-next.config.ts` (configuring the static-assets incremental cache — see design.md Decision 5), a dev-bindings init call in `next.config.ts`, and `@opennextjs/cloudflare` + `wrangler` as dependencies, plus a `preview` package.json script for local verification (`opennextjs-cloudflare build && opennextjs-cloudflare preview`, which populates the cache — a raw `wrangler dev` against an unpopulated cache gives false negatives for static routes).
- **New capability**: `cloudflare-deployment-compat` — codifies the constraints this change satisfies (no runtime `node:fs` reads of bundled application data in any genuinely dynamic Worker-executed code path, no Node.js-runtime Proxy/middleware, static routes served from a populated build-time cache) so future changes don't silently reintroduce them.
- **Out of scope**: actually running `wrangler deploy` against a live Cloudflare account/domain, DNS cutover, or provisioning Cloudflare-side secrets — those are real, external, billing/downtime-relevant actions taken as an explicit follow-up once this change is merged and re-verified, not part of this change.

## Capabilities

### New Capabilities
- `cloudflare-deployment-compat`: the constraints the codebase must satisfy to remain deployable to Cloudflare Workers via the OpenNext adapter — no request-time `node:fs` reads of bundled application data in genuinely dynamic routes, no Node.js-runtime Proxy/middleware, and prerendered routes served from a populated build-time cache rather than re-executed per request.

### Modified Capabilities
- `admin-access-gate`: the owner-only `/admin` gate changes from HTTP Basic Auth (401 challenge via `proxy.ts`) to a cookie-session login (login form → Route Handler → signed httpOnly cookie checked by a protected layout). Credential source, timing-safe comparison, rate limiting, fail-closed behavior, robots exclusion, and "no effect on the public site" all carry forward unchanged — the challenge/session mechanism changes, and the admin area additionally stops rendering the public site's marketing chrome (hero, chat widget, footer), rendering under its own minimal root layout instead.

## Impact

- **Modified files**: `lib/rag/retrieve.ts` (dynamic import instead of `readFileSync`), `app/api/chat/route.ts` (awaits `loadIndex()`; error helpers use the new site-config artifact instead of `getProfile()`), `lib/rag/eval-run.ts` (awaits `loadIndex()`), `package.json` (next version bump, new deps, new scripts, prebuild wiring), `next.config.ts` (OpenNext dev-bindings init). `app/opengraph-image.tsx` is unchanged.
- **Moved files**: `app/layout.tsx` → `app/(marketing)/layout.tsx`; `app/page.tsx` → `app/(marketing)/page.tsx`; `app/admin/page.tsx` → `app/admin/(protected)/page.tsx` (updated to read `chapterIds` from the site-config artifact instead of calling `getExperiences()`).
- **New files**: `wrangler.jsonc`, `open-next.config.ts`, `app/admin/layout.tsx` (new independent root layout, minimal chrome), `app/admin/(protected)/layout.tsx` (session check + redirect), `app/admin/login/page.tsx` (login form), `app/admin/login/route.ts` (credential verification + cookie), `lib/admin/session.ts` (cookie signing/verification), `lib/admin/credentials.ts` (renamed/refactored from `basicAuth.ts`), a small build-time script generating the `{ contact, chapterIds }` site-config JSON, tests for all of the above.
- **Removed files**: `proxy.ts`, `lib/admin/basicAuth.ts` (replaced by `credentials.ts`), and their now-superseded tests.
- **Unchanged**: `lib/content/read.ts` and its entire existing fixture-based test suite (still `readFileSync`-based, still correct for every build-time/static consumer — `app/(marketing)/page.tsx`, `app/(marketing)/layout.tsx`, `components/StructuredData.tsx`, all prerendered), `lib/admin/credentials.ts`'s timing-safe comparison logic (carried over from `basicAuth.ts`, not the HTTP-challenge wrapper around it), `lib/chat/rateLimit.ts` (reused as-is for the login route), the admin dashboard's Neon reports code, and every other accepted capability (chat, hero, analytics) — none of their observable behavior changes.
- **No change** to `content-indexing-pipeline` or `retrieval-grounded-generation` requirements (build-time index generation and retrieval/generation logic are untouched; only the in-memory loading mechanism changes).
- **Verification**: re-run the same real build + `wrangler dev` check used in both spikes (not just `npm test`) to confirm `/api/chat` streaming and its error paths, `/opengraph-image`, `/admin`'s full login-to-dashboard flow, and Neon writes all work end-to-end under the adapter before this is considered done.
