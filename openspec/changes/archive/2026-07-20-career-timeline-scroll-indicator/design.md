## Context

JOS-56 shipped `CareerTimeline` as a server component deliberately (design.md decision: "no new client component upgrade needed... avoids shipping unnecessary JS"), because navigation alone only needs real anchor links. JOS-57 asks for something inherently JS-only — knowing which chapter is scrolled into view — so that prior decision needs to be revisited for this component specifically. The chapters it needs to track already expose stable `id`s (`experience.id`, added in JOS-56's modification to `career-chapter-rendering`), but they're rendered by a sibling component (`CareerChapters`), not by `CareerTimeline` itself — there's no parent/child or context relationship between them today.

## Goals / Non-Goals

**Goals:**
- The timeline node for the chapter currently in view is visually and semantically marked as current (`aria-current="location"`).
- The indicator updates correctly both from free scrolling and from clicking a timeline node (AC1 and AC2 are the same underlying mechanism — no special-casing needed for navigation-triggered updates).
- Any visible transition accompanying an indicator update is suppressed under `prefers-reduced-motion` (AC3).
- No-JS behavior is unaffected: navigation (JOS-56) still works with JavaScript disabled; only the position indicator is unavailable, which is inherent to what this feature is (there's no meaningful "current scroll position" without JS).

**Non-Goals:**
- No changes to the chapters' own content or rendering.
- No new dependency — native `IntersectionObserver`, no scroll-event polling, no Framer Motion (this is a CSS color/border transition, not a JS-computed animation, so `useReducedMotion` isn't needed here — Tailwind's `motion-safe:` variant is the simpler, sufficient tool).

## Decisions

- **`IntersectionObserver` for the general case, with a small, deliberate exception (see below).** The primary tracking mechanism is still `IntersectionObserver` — no manual scroll-position math for the common case, and the browser batches updates cheaply even with more chapters later.
- **Look up chapter elements by id (`document.getElementById(experience.id)`) inside a `useEffect`, rather than lifting shared state up into a common parent.** `CareerTimeline` and `CareerChapters` are independent siblings mounted directly in `app/page.tsx`; introducing a shared parent/context just for this would be a bigger structural change than a JS-budget-conscious "S" story warrants. The two components already share `experience.id` as their coupling key (the `#{id}` href JOS-56 established) — this reuses that same key rather than inventing a new one.
- **`rootMargin: "-20% 0px -35% 0px"` on the observer** — a reading-line band through the viewport's 20%–65% range. Chosen by computing real `getBoundingClientRect()` overlap against the live app's actual chapter geometry (rather than guessing), after two narrower bands (a 10%-tall and a 15%-tall band centered lower) were tried and calculated to never overlap this page's actual chapter elements across its achievable scroll range.
- **A dedicated scroll listener handles the case `IntersectionObserver` fundamentally cannot: a trailing chapter whose element never crosses the band at all, because the page has no more room to scroll.** `IntersectionObserver` callbacks fire only on a state *change* — if a short chapter's rect sits entirely below the band for the page's *entire* scrollable range (verified with this app's real content: a 2-chapter, collapsed-by-default page is short enough that this genuinely happens), the callback for it never fires, band width notwithstanding. A minimal `scroll` listener (`{ passive: true }`, no debouncing) checks `scrollY + innerHeight >= scrollHeight - 2` and, when true, unconditionally activates the last chapter. This is a deliberate, narrow exception to "no manual scroll-position math," added only after live testing surfaced the gap, not a wholesale return to scroll-based tracking.
- **Both the `IntersectionObserver` callback and the scroll listener call the same `updateActiveId()` function, rather than each calling `setActiveId` independently.** The "at bottom" check runs first in that shared function and returns early when true, so whichever listener fires last for a given scroll gesture, the outcome is identical — no race between "IntersectionObserver says chapter N" and "scroll listener says the last chapter" depending on firing order.
- **Only update the active id when a chapter is found in the current intersecting-id set (populated additively/subtractively per entry, not per callback batch); a callback with nothing currently intersecting leaves the active id untouched.** This is what makes the "stays current until a different chapter intersects" behavior work for the normal (not-at-bottom) case — no explicit "clear" logic needed.
- **`CareerTimeline` becomes `"use client"`.** It's still server-rendered for first paint (Next.js server-renders client components too — `"use client"` only adds hydration, it doesn't remove SSR), so this doesn't reopen the no-JS requirement JOS-56 already satisfied for navigation; it only makes the new position-tracking enhancement client-only, which is unavoidable for a feature that is inherently about live scroll state.
- **Style the active state with Tailwind's `motion-safe:` variant**, not a JS-based `prefers-reduced-motion` check (unlike `HeroFramer`, which needed `useReducedMotion` because it's driving a JS-computed animation via Framer Motion). Here the "animation" is just a CSS `transition-colors` on a class/attribute change — `motion-safe:transition-colors` is the direct, simpler equivalent, resolved natively by the browser with no JS involved in the reduced-motion decision itself.
- **While touching `timelineNodeClass`'s existing `transition-colors` (currently unconditional, added in JOS-56 for hover), change it to `motion-safe:transition-colors`.** It's the same Tailwind utility the new active-state color change would also rely on, so leaving it unconditional would mean hover transitions still animate under reduced motion while the new active-indicator transition doesn't — an inconsistency on the same class. Gating both is a small, directly-related correction, not unrelated scope creep.

## Risks / Trade-offs

- [Risk] `document.getElementById` lookups run once per `experiences` change in a `useEffect`, after `CareerChapters` has had a chance to render — if `CareerTimeline` somehow mounted and ran its effect before `CareerChapters`' DOM existed, elements could be missed → Mitigation: both components render synchronously in the same server-rendered HTML payload (no client-side data fetching or lazy mounting involved), so by the time any client component's effects run post-hydration, the full DOM — including `CareerChapters`' elements — is already present.
- [Risk] Rapid scrolling could fire many observer callbacks → Mitigation: `IntersectionObserver` is browser-batched and threshold-based already (not per-scroll-pixel), and each callback here just does an id comparison and, at most, one `setState` — cheap regardless of scroll speed.
- [Risk] `IntersectionObserver` callback delivery is throttled to zero in a backgrounded/non-focused browser tab (Chrome deprioritizes rendering-pipeline-tied observers for hidden tabs) — confirmed live: even observing `document.body` with no `rootMargin` at all never fired a callback in the automated verification tab (`document.hidden: true`), matching the same root cause already documented for `requestAnimationFrame` throttling in `hero-reduced-motion-alternative`'s 60fps measurement → Mitigation: this only affects this automation environment's ability to *verify* the feature live, not real usage — a real visitor scrolling the page has it focused/visible by definition. The plain `scroll` listener (unrelated to the rendering pipeline) fires normally regardless of tab visibility and was what was actually confirmed working live; the `IntersectionObserver`-driven mid-page tracking is verified instead by the jsdom test suite, which invokes the callback directly and doesn't depend on real browser observer delivery.
