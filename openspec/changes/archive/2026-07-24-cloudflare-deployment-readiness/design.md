## Context

The owner wants to host CareerDNA on Cloudflare (existing home for other projects) instead of Vercel. A hands-on spike — a real `@opennextjs/cloudflare` build plus `wrangler dev` run against a throwaway copy of the repo, using real `.env.local` credentials against the real Neon database (test row cleaned up afterward) — found:

- `@neondatabase/serverless`'s HTTP driver works with no changes (`/api/events` → `204`, no errors).
- `app/api/chat/route.ts`'s `loadIndex()` (`lib/rag/retrieve.ts`) throws `[unenv] fs.readFileSync is not implemented yet!` — the adapter's own Node-compat shim doesn't implement it, contradicting Cloudflare's generic `nodejs_compat` docs (which describe a `/bundle` virtual fs) for this adapter version.
- `app/opengraph-image.tsx` hits the identical error via `next/og`'s default system-font loading. Confirmed via `.open-next/assets` that this route is not pre-baked into a static asset by the adapter despite Next's own `○ Static` build-output label — the fix is required, not optional.
- The build fails outright with `Node.js middleware is not currently supported` — Next 16's Proxy (used by `proxy.ts` for `/admin`'s HTTP Basic Auth, per the accepted `admin-access-gate` capability) always runs on the Node.js runtime, which the adapter currently rejects. This is a live, unresolved gap (`cloudflare/workers-sdk#13755`); reverting to the legacy Edge `middleware.ts` convention doesn't reliably dodge it either, per that same thread.
- `@opennextjs/cloudflare@1.20.2` requires `next@">=16.2.11"`; the repo pins `16.2.10`.

The owner chose (of three options presented — cookie-session login, Route-Handler-rendered Basic Auth, or moving the gate to Cloudflare Access) to replace Basic Auth with a cookie-session login form, since Route Handlers (unlike Proxy) build and run fine under the adapter, and this keeps the gate as application code rather than adding a new Cloudflare Zero Trust dependency.

**A second round of verification**, done after starting implementation (rebuilding the pre-fix code against a fresh spike copy, specifically exercising `/admin` and `/api/chat`'s error paths rather than only its success path), found the identical `fs.readFileSync not implemented` error is not limited to the RAG index:

- `/admin`'s dashboard page calls `getExperiences()` (`lib/content/read.ts`) directly at request time (the page is `force-dynamic`, for fresh Neon queries).
- `/admin` also renders inside the shared root layout, which — since the hero-laptop and chat-greeting stories (JOS-90/91) — calls `getProfile()`/`getFaq()` for the hero terminal text and chat widget copy. Since `/admin` is dynamic, that layout re-executes at request time for admin requests too, hitting the same error.
- `/api/chat`'s rate-limited/unavailable error responses call `getProfile()` solely to include the contact email/scheduling link in the JSON error body.

`lib/content/read.ts` is a much more widely-used, foundational module than the RAG index (used by every page, including static ones, and extensively exercised by a `contentRoot`-parameterized fixture test suite). Presented with the fork this creates (patch the shared layout to also avoid runtime reads for `/admin`'s benefit, vs. give `/admin` its own layout and stop precomputed-content-threading through the shared one, vs. rework `lib/content/read.ts` globally), the owner chose to give `/admin` its own independent, minimal root layout — both fixing the crash and removing marketing chrome that likely never belonged on an internal dashboard.

## Goals / Non-Goals

**Goals:**
- Make the existing build produce a working Cloudflare Workers bundle via OpenNext, verified the same way the spike did (real build + `wrangler dev`, not just unit tests).
- Preserve every other capability's behavior exactly — chat, hero, analytics, content, SEO metadata — only the admin auth mechanism and the RAG-index loading mechanism change.
- Codify the constraints found (no request-time `fs` reads in genuinely dynamic routes, no Node-runtime Proxy, static routes served from a populated cache) as an accepted spec so future work doesn't silently reintroduce them.

**Non-Goals:**
- Actually running `wrangler deploy` against a live Cloudflare account, DNS cutover, or provisioning Cloudflare-side secrets — real, external, billing/downtime-relevant actions taken as an explicit follow-up once this change is verified, not part of it.
- Any change to chat behavior, retrieval/generation logic, analytics schema, or the hero/chat UI work already shipped.
- Migrating incremental-cache/ISR to R2 — not needed; this site has no ISR-revalidated routes (see Decision 5, which configures the simpler read-only static-assets cache instead).

## Decisions

### 1. RAG index: dynamic `import()` instead of runtime `readFileSync`
`lib/rag/embed.ts`'s build step continues to write `lib/rag/index.json` exactly as today (`content-indexing-pipeline` capability unchanged). `lib/rag/retrieve.ts`'s `loadIndex()` changes from `readFileSync(path)` to `await import("./index.json", { with: { type: "json" } })`, becoming `async`; its one call site (`app/api/chat/route.ts`) awaits it. This bundles the data at build time via the module graph instead of reading it from a virtual filesystem at request time — sidesteps the adapter's `fs` gap entirely rather than depending on it being fixed upstream. `path` becomes unused/removed (a dynamic-import specifier has to be statically analyzable for the bundler to include the right file, so an arbitrary caller-supplied path no longer makes sense). **Alternative considered — a static top-level `import` instead of a dynamic one:** rejected once tried, because a static import forces `lib/rag/index.json` to exist merely to import *anything* from `retrieve.ts` — which would break the existing `retrieve.test.ts` (which exercises `cosineSimilarity`/`retrieveTopK` and never calls `loadIndex()`) in any environment where the index hasn't been generated yet. The dynamic import preserves today's laziness: only calling `loadIndex()` requires the file, exactly as the old `readFileSync` did. **Alternative considered — waiting for/relying on the adapter's `/bundle` fs support:** rejected, since the spike proved it doesn't currently work for this call regardless of what the generic docs say.

### 1a. Site config (`contact` + `chapterIds`): the same dynamic-import pattern, for the two remaining request-time content reads
`/api/chat`'s error responses and `/admin`'s dashboard each need one small, build-time-stable slice of content at request time — the contact info, and the ordered chapter-ID list — without pulling in the rest of `lib/content/read.ts`'s `readFileSync`/`readdirSync` machinery. A small build-time script (run as part of the same `prebuild` step, using `lib/content/read.ts`'s existing `getProfile()`/`getExperiences()` — fine, since that runs in the real build-time Node process, not the Worker) writes a tiny generated JSON (`{ contact, chapterIds }`). The two call sites load it the same way as the RAG index — a dynamic `import()`, not `readFileSync`. **Alternative considered — rework `lib/content/read.ts` itself to load from one generated JSON globally (mirroring the RAG index exactly for every consumer):** rejected — `getExperiences()`/`getSkills()`/`getProjects()` all accept an optional `contentRoot` parameter that the existing test suite (`makeFixtureRoot()`-based fixtures across `lib/content/*.test.ts`) relies on to test against arbitrary temp directories; a dynamic-import specifier can't be parameterized by a runtime string, so this would force rewriting that entire fixture-based test architecture for no behavioral benefit — every *other* consumer of `lib/content/read.ts` (`app/(marketing)/page.tsx`, `app/(marketing)/layout.tsx`, `components/StructuredData.tsx`) renders on static, prerendered routes and never executes in the Worker at all. Scoping the fix to exactly the two dynamic call sites is narrower and lower-risk.

### 1b. `/admin` gets its own independent root layout
`app/layout.tsx` moves to `app/(marketing)/layout.tsx`, and `app/page.tsx` to `app/(marketing)/page.tsx` — a route group, so the public URL structure (`/`) is unchanged. `app/admin/layout.tsx` becomes a second, independent root layout (its own `<html>`/`<body>`, per Next.js's documented multiple-root-layouts pattern) with no hero, chat widget, footer, or structured data — just the admin surface. Route Handlers (`app/api/**`) and metadata-generating files (`robots.ts`, `sitemap.ts`, `opengraph-image.tsx`, `favicon.ico`) don't render through a layout at all, so this split doesn't touch them. **Alternative considered — keep `/admin` inside the shared layout and instead have that layout's `getProfile()`/`getFaq()` calls also read from the Decision 1a site-config artifact:** rejected — it would mean the marketing root layout's data shape is partly dictated by what a completely unrelated internal dashboard needs, and it leaves recruiter-facing chrome (the chat widget, the hero) on the owner's private analytics page, which the owner confirmed was never intentional.

### 2. OpenGraph image: no code change needed, once the cache is correctly configured
`app/opengraph-image.tsx` is left exactly as it was. It first *looked* like it needed the standard Next.js/Vercel "fetch an explicit font via `new URL(..., import.meta.url)`" pattern (used for edge-deployed `ImageResponse` elsewhere), since it threw the identical `fs.readFileSync not implemented` error in an early test. Two things resolved this without a code change: (1) that early test used a raw, unpopulated `wrangler dev` invocation — a false-negative testing setup, see Decision 5; with the static-assets cache correctly populated, this route renders once during `next build`'s own Node.js prerendering (real filesystem access, no problem) and is served from that cached output thereafter, never re-executing in the Worker. (2) **The fetch-based font pattern was tried anyway, as a defense-in-depth measure, and it broke `next build` itself** — `fetch(new URL(..., import.meta.url))` for a local file isn't supported by `next build`'s prerendering step under this route's (default, Node.js) runtime target; it's a pattern that depends on bundler-specific asset-URL rewriting that isn't applied here, unrelated to Cloudflare at all. It was reverted. **Alternative considered:** declare the route `runtime = "edge"` to get that bundler treatment — rejected without testing further, since the simpler fact (this route is cache-served, not live-executed) already makes the whole font concern moot, and forcing Next's classic Edge Runtime here would trade one set of untested constraints for another with no benefit.

### 3. Admin auth: cookie-session login replacing Basic Auth
- **Routing**: nested inside `app/admin/`'s own independent root layout (Decision 1b): the dashboard moves to `app/admin/(protected)/page.tsx`, with `app/admin/(protected)/layout.tsx` performing the session check via `cookies()` (`next/headers`) and `redirect("/admin/login")` on missing/invalid/expired session — route groups don't affect the URL, so `/admin` still resolves to the dashboard. `app/admin/login/page.tsx` (GET, renders a plain username/password form) and `app/admin/login/route.ts` (POST, handles submission) sit outside the protected group, both still under `app/admin/layout.tsx`'s minimal chrome.
- **Credential check**: `lib/admin/basicAuth.ts` is refactored to `lib/admin/credentials.ts`, keeping the exact `constantTimeEquals` byte-folding comparison but exposing `verifyCredentials(user, pass, expected)` directly — the Basic-Auth-specific header parsing/base64-decoding is deleted since there's no `Authorization` header to parse anymore (the login route reads form fields instead).
- **Rate limiting**: the login route reuses `checkRateLimit`/`createUpstashRateLimitStore` from `lib/chat/rateLimit.ts` unchanged, keyed on IP, with the same fail-open-on-backend-error behavior — same mechanism as before, just invoked from a Route Handler instead of Proxy.
- **Session token**: `lib/admin/session.ts` issues an HMAC-SHA256-signed token (issued-at + expiry + signature) via Web Crypto's `crypto.subtle` — not `node:crypto` — so the same code runs identically on Workers, Node, and in tests without a runtime-specific branch (mirrors `basicAuth.ts`'s existing `atob`-not-`Buffer` convention for the same reason). **Signing secret**: reuses `ADMIN_PASSWORD` as the HMAC key rather than introducing a new required environment variable — if that secret is compromised the gate is already compromised, so a separate session secret buys no real additional security here, and avoids a new manual-setup step in `README.md`'s existing "one-time manual setup" list (Upstash, Neon, site URL). **Alternative considered:** a dedicated `ADMIN_SESSION_SECRET` — rejected as unnecessary operational overhead for a single-owner admin gate at this equivalent trust level.
- **Cookie attributes**: `HttpOnly`, `Secure`, `SameSite=Strict`, `Path=/admin` (never sent on public routes, satisfying the accepted capability's "no cookie on the public site" purpose, which is about the *public* site, not the admin area itself), `Max-Age` matching the token's embedded expiry (7 days, matching typical low-traffic owner-dashboard session length — not a sensitive value, revisit if it feels wrong in practice).
- **Robots/indexability**: `/admin/login`'s page metadata gets the same `robots: { index: false, follow: false }` the dashboard already declares; `lib/seo` robots policy gains the login path alongside the existing admin disallow (or the existing `/admin/:path*`-style disallow already covers it — confirmed during Tasks).

### 4. Dependency and tooling additions
`next` bumps from `16.2.10` to the exact `16.2.11` (latest available 16.2.x patch, the adapter's stated minimum) — a plain patch bump, not expected to interact with the existing `typescript@5.9.3` pin note (that note is about Next↔TypeScript build-tooling compatibility, unaffected by a Next patch bump within the same minor). `@opennextjs/cloudflare` and `wrangler` are added as dependencies; `wrangler.jsonc` (compatibility date ≥ `2025-09-01` per Cloudflare's `node:fs` support threshold, `nodejs_compat` flag, `assets` binding pointing at `.open-next/assets`) and `open-next.config.ts` (see Decision 5) are added at the repo root; `next.config.ts` gains the `initOpenNextCloudflareForDev()` call for local-binding parity in `next dev`; `package.json` gains a `preview` script (`opennextjs-cloudflare build && opennextjs-cloudflare preview`) for the real local verification loop.

### 5. Incremental cache: static-assets override, and `preview`/`populateCache` as the only valid local verification tool
A third round of empirical verification (checking whether `/` itself — not just `/admin`/`/api/chat` — survives a real build) initially looked catastrophic: with `open-next.config.ts`'s default (unconfigured) incremental cache, raw `wrangler dev` returned `500` with the identical `fs.readFileSync not implemented` error for `/`, `/opengraph-image`, *and* `robots.txt`/`sitemap.xml` succeeded — meaning even genuinely static, content-free routes appeared to work while any content-touching route failed, regardless of Next's own `○ Static` labeling. This traced to a real, distinct gap: the adapter ships a `static-assets-incremental-cache` override (`@opennextjs/cloudflare/overrides/incremental-cache/static-assets-incremental-cache`) specifically for static-first sites, which serves prerendered build-time output directly, bypassing the server function entirely — but it requires (a) being explicitly configured in `open-next.config.ts` (`incrementalCache: staticAssetsIncrementalCache, enableCacheInterception: true`) and (b) the cache actually being *populated* before serving, via `opennextjs-cloudflare preview` (which runs cache population automatically) or `opennextjs-cloudflare populateCache local` directly — plain `wrangler dev` against the raw build does neither, silently falling through to a live re-render of every route on every request, which is what produced the false alarm. With both configured and populated, `/` and `/opengraph-image` return correct `200`s and valid content, while `/admin` (genuinely `force-dynamic`, correctly bypassing this read-only cache) still fails exactly as expected until Decisions 1a/1b/3 are applied. **Consequence for this change:** `open-next.config.ts` must configure the static-assets cache override (not the empty default), and all Cloudflare-specific verification (Tasks §8) must use `opennextjs-cloudflare preview` (or `build` + `populateCache local` + `wrangler dev`), never a raw `wrangler dev` invocation against an unpopulated cache — the latter gives false negatives for genuinely static routes. **Alternative considered:** the R2-backed incremental cache (supports revalidation) — rejected as unnecessary; this site has no ISR routes, so the simpler read-only static-assets cache is a strictly better fit and needs no R2 bucket provisioned.

## Risks / Trade-offs

- **[Risk]** The OpenNext Cloudflare adapter is actively evolving (the Proxy/middleware gap could be fixed upstream, or new gaps could appear in a future version). → Mitigation: the new `cloudflare-deployment-compat` spec documents the current constraints so a future upgrade attempt has a clear checklist to re-verify against, rather than rediscovering these the hard way again.
- **[Risk]** Reusing `ADMIN_PASSWORD` as the session-signing key means rotating the password invalidates all sessions (forces re-login) — acceptable for a single-owner dashboard, and arguably a feature (rotating the password should invalidate stale sessions).
- **[Trade-off]** Losing the native browser Basic-Auth popup for a custom login form is a real UX change, explicitly chosen by the owner over the alternatives (Route-Handler-rendered Basic Auth, Cloudflare Access) to keep the gate as portable application code.
- **[Risk]** If a future deploy skips or delays cache population (e.g. `wrangler deploy` without an equivalent `populateCache remote` step), the *first* production request to a static route could still hit a live re-render and crash on the same `fs` issue, until the cache is populated. → Mitigation: Tasks §8 verification explicitly checks that population happens as part of the deploy workflow, not left as an undocumented manual step; flagged as a real production-readiness item, not assumed safe from local testing alone.

## Migration Plan

Branch → fix `loadIndex()` (dynamic import) → add the site-config build artifact → add OG image fonts + verify → split root layouts (`(marketing)` group + independent `app/admin/layout.tsx`) → refactor admin auth (credentials.ts, session.ts, login route/page, protected route group, delete `proxy.ts`) → bump `next` to `16.2.11` → add Cloudflare tooling files (including the static-assets incremental cache config, Decision 5) → `npm test` / `tsc` / `validate:content` clean → real verification: `npx opennextjs-cloudflare build` + `opennextjs-cloudflare preview` (never a raw, unpopulated `wrangler dev`), hit the landing page, `/api/chat` streaming *and* its error paths, `/opengraph-image`, `/api/events` (Neon), `/admin` with no session, and the full login → dashboard flow → merge → sync spec + archive. Rollback: revert the branch; `proxy.ts`-based Basic Auth, the shared admin layout, and the runtime `fs` reads are all restorable from git history if Cloudflare is abandoned, though at that point Vercel (this app's zero-adapter-friction alternative, discussed and available if this path stalls) remains the fallback.
